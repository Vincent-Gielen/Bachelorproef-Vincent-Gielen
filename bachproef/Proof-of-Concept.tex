\section{PoC 1: Gebruik maken van  een API}
In deze PoC wordt er getracht om via het mainframe contact te leggen met een API. Er wordt contact gelegd met een private API die op hetzelfde mainframe draait, alsook de publieke API van \href{https://github.com/open-meteo/open-meteo}{open-meteo} om het weer in Madrid te verkrijgen. Zo kan er gekeken worden of er een verschil is als een API op een mainframe draait of niet, en welke parameters daarvoor moeten worden veranderd.

\subsection{Python}
In de code van de Python bestanden \ref{mint:privateAPIPython} en \ref{mint:publiekeAPIPython} is het proces bijna hetzelfde. De \texttt{requests} package van Python wordt geïmporteerd. Deze veelgebruikte package zorgt ervoor dat API requests op een simpele manier kunnen worden uitgevoerd. Voor de private API moet er een gebruikers-ID en een wachtwoord worden meegegeven om juist te kunnen aanmelden op de het mainframe. Hiervoor wordt een extra package van Python gebruikt. Voor de publieke API moeten er bepaalde parameters worden meegegeven om de informatie te verkrijgen over een specifieke locatie (hier Madrid). Bij het gebruik van de requests package wordt automatisch geverifieerd of de server (het systeem dat wordt aangesproken) wel veilig is. Dit wordt gedaan aan de hand van internationale standaarden en certificaten. Bij de private API is dit manueel uitgezet, omdat we het systeem kennen, en de certificaten dit voorbeeld onnodig complex zouden maken. Nadat de request al dan niet succesvol is verstuurd, wordt er een response ontvangen. Als dit antwoord goed is (met de code 200), dan worden de resultaten afgebeeld op het scherm.

\begin{listing} [h]
    \begin{minted}[frame=lines, framesep=2mm, linenos]{python}
        # De Python package requests en getpass worden geimporteerd
        import requests
        from getpass import getpass
        
        # De credentials om in te loggen worden aan de gebruiker gevraagd en opgeslaan
        myCreds = ('','')
        
        if myCreds[0] == '':
            userid = input("Give your userid: ")
            myCreds = (userid,'')
        
        if myCreds[1] == '':
            passwd = getpass("Give your password: ")
            myCreds = (myCreds[0], passwd)
        
        # De URL naar de private API van mainframeyin
        url = "https://mainframeyin:8092/LWWAPI/API/API"
        
        # De credentials worden meegegeven zodat correct kan worden aangemeld
        # Er wordt niet geverifieerd of deze API veilig is, omdat we deze zelf beheren
        response = requests.get(url, auth=myCreds, verify=False)
        
        if (response.status_code != 200):
            raise Exception(f"{response.status_code} - {response.text}")
        else:
            # We verkrijgen de gegevens in JSON formaat
            ApiList = response.json()
        
        # De verschillende API namen worden afgebeeld
        for api in ApiList['API']:
            print(api['APIName'])
    \end{minted}
  \caption[Private API op hetzelfde mainframe in Python]{\label{mint:privateAPIPython}Contact leggen met een private API op hetzelfde mainframe met Python}
\end{listing}

\begin{listing} [h]
    \begin{minted}[frame=lines, framesep=2mm, linenos]{python}
        # De Python package requests wordt geimporteerd
        import requests
        
        # De coördinaten voor Madrid
        params = {
            "latitude": 40.4168,
            "longitude": -3.7038,
            "current_weather": "true",
            "temperature_unit": "celsius"
        }
        
        # De URL naar de publieke API van open-meteo
        url = "https://api.open-meteo.com/v1/forecast"
        
        # Er zijn hier geen credentials nodig, aangezien de API public beschikbaar is
        # We geven de parameters mee, zodat we het weer van Madrid krijgen
        response = requests.get(url, params=params)
        
        # Als er een error is, wordt die hier getoond
        if response.status_code != 200:
            raise Exception(f"{response.status_code} - {response.text}")
        else:
            # We verkrijgen de gegevens in JSON formaat
            weather_data = response.json()
        
        # We halen enkele gegevens uit de JSON
        current = weather_data['current_weather']
        temp = current['temperature']
        wind = current['windspeed']
        
        # Het resultaat wordt afgebeeld
        print(f"Het weer in Madrid:")
        print(f"Temperatuur: {temp} graden Celsius")
        print(f"Windsnelheid: {wind} km/h")
    \end{minted}

\subsection{Rexx}
In de code van de Rexx bestanden \ref{mint:privateAPIRexx} en \ref{mint:publiekeAPIRexx} wordt gebruik gemaakt van het Unix commando \texttt{cURL} (Client URL). Dit commando wordt veel gebruikt voor het communiceren tussen systemen en is dus niet specifiek voor Rexx. Oorspronkelijk zou er voor Rexx gebruik gemaakt worden van de z/OS Client WET (Web Enablement Toolkit), maar dit bleek te complex voor het relatief kleine voorbeeld. Ervaren systeemprogrammeur Lionel Dyck gaf dit advies:

\blockquote[Lionel B. Dyck - IBM Z Champion on Discord]{I once looked at WET and, tbh(to be honest), gave up - seemed too complex for what it does.  Using bpxwunix with standard unix tools seemed easier/simpler and more maintainable down the road.  Just my \$0.01 - ymmv(your mileage may vary)}

Het algemene proces in Rexx is hetzelfde als in Python. Er wordt een request gestuurd naar een API via een url, met enkele parameters of credentials. In de Rexx code moeten de parameters en credentials op handmatige manier toegevoegd worden aan de url voordat we deze kunnen versturen met het curl commando. Het curl commando kan op verschillende manieren worden aangepast aan de noden van de use case, gelijkaardig aan de requests package van Python. Er wordt een stem variabele meegegeven, dit werkt op een gelijkaardige manier als een array, waar de response zal in terechtkomen. Nadat we checken op errors (rc = 0), kunnen we in de response met handige string manipulatie de gewenste resultaten afbeelden.

\begin{listing} [h]
    \begin{minted}[frame=lines, framesep=2mm, linenos]{python}
        /* REXX */
        
        /* De URL naar de private API */
        url = 'https://mainframeyin:8092/LWWAPI/API/API'
        
        /* De credentials om in te loggen aan de gebruiker vragen */
        /* We gebruiken PARSE PULL zodat hoofdletters behouden blijven */
        say "Give your userid: "
        parse pull userid
        myCreds.1 = userid
        
        say "Give your password: "
        parse pull passwd
        myCreds.2 = passwd
        
        /* We bouwen het curl commando op:
        -k : --insecure (vergelijkbaar met verify=False in Python)
        -u : --user (voor Basic Authentication)
        */
        commando = "curl -k -u "myCreds.1":"myCreds.2" '"url"'"
        
        /* Voer het Unix commando uit */
        rc = bpxwunix(commando,,response.,error.)
        
        /* Als er een error is, wordt die hier getoond */
        if rc <> 0 then do
            say "Error executing curl. Return code:" rc
            do i = 1 to error.0
                say error.i
            end
            exit 8
        end
        
        /* Verwerking van de JSON response */
        /* We zoeken naar de APINames in de JSON */
        ApiList = response.1
        
        say "Lijst met API's:"
        do while pos('"APIName":"', ApiList) > 0
            /* De verschillende API namen worden afgebeeld */
            parse var ApiList '"APIName":"' name '"' ApiList
            say name
        end
        
        exit 0
        
    \end{minted}
    \caption[Private API op hetzelfde mainframe in Rexx]{\label{mint:privateAPIRexx}Contact leggen met een private API op hetzelfde mainframe met Rexx}
\end{listing}

\begin{listing} [h]
    \begin{minted}[frame=lines, framesep=2mm, linenos]{python}
        /* REXX */
        
        /* We moeten de parameters in de url steken van de publieke API */
        url = 'https://api.open-meteo.com/v1/forecast?latitude=40.4168' || ,
        '&longitude=-3.7038&current_weather=true&temperature_unit=celsius'
        
        /* We maken het Unix commando met de publieke API van open-meteo*/
        commando = "curl '"url"'"
        
        /* Dit is een Rexx commando (bpxwunix) om een Unix commando (curl)
        uit te voeren. Het antwoord komt in response. terecht, enige errors in error.*/
        rc = bpxwunix(commando,,response.,error.)
        
        /* Als er een error is, wordt die hier getoond */
        if rc <> 0 then do
            say "Error executing curl. Return code:" rc
            do i = 1 to error.0
                say error.i
            end
            exit 8
        end
        
        /* We zoeken specifiek binnen de 'current_weather' sectie
        We snijden de tekst van links naar rechts
        op zoek naar temperature en windspeed */
        parse var response.1 'current_weather":{' data '}' .
        parse var data 'temperature":' temp ','
        parse var data 'windspeed":' wind ','
        
        say "Het weer in Madrid:"
        say "Temperatuur: " temp "graden Celsius"
        say "Windsnelheid:" wind "km/h"
        
        exit 0
        
    \end{minted}

\end{listing}

\subsection{Bevindingen}
Eerst en vooral lijkt er geen verschil te zijn tussen het gebruik van een API op het (eigen) mainframe, of buiten het mainframe. Dit komt omdat de connectie hoe dan ook via http gebeurt, dus elke verbinding wordt als een verbinding van buiten beschouwd. Voor het aanmaken van de bestanden en schrijven en uitvoeren van de code waren er problemen met de tags van de files. Voor het correct interpreteren en uitvoeren van de Python en Rexx bestanden is een verschillende filetag nodig. Zo'n filetag kan worden veranderd met behulp van het \texttt{chtag} commando. Voor Python werd \texttt{chtag -tc 819} gebruikt, en voor Rexx \texttt{chtag -tc 1047}. Om bestaande inhoud van een bestand om te zetten naar een nieuw bestand met de juiste tag kan gebruik worden gemaakt van het \texttt{iconv} commando. Daarnaast moest voor het uitvoeren van de Rexx bestanden via USS het commando \texttt{chmod +x bestandsnaam.rexx} worden uitgevoerd. Bovenstaande bevindingen hebben, behalve een eventuele deuk in de gebruiksvriendelijkheid van het mainframe, niks te maken met de hierop volgende requirements.

\begin{itemize}
\item \textbf{Integratie met tools op het mainframe:}
Rexx levert een zeer natuurlijke integratie met het mainframe-ecosysteem door het gebruik van \texttt{bpxwunix}, waarmee direct gebruik kan worden gemaakt van de onderliggende Unix System Services (USS). Ook moet worden gezegd dat het gebruik van WET eventueel een nog betere integratie met Rexx zou kunnen geven. Python op het mainframe vereist enkele specifieke acties, zoals een correct geconfigureerde \texttt{venv} (virtual environment) en de installatie van specifieke libraries. Die laatste kunnen eventueel moeilijk zijn als bepaalde packages niet worden toegelaten. In dit geval was dat geen probleem.

\item \textbf{Integratie met tools buiten het mainframe:}
Python is hier de duidelijke winnaar bij het gebruiken van API's. Dankzij de \texttt{requests} package, die specifiek is ontworpen voor moderne webcommunicatie en helpt met zaken als SSL-verificatie en JSON-parsing, kan Python zonder problemen communiceren met een externe API. Rexx moet voor externe integratie vertrouwen op tools zoals \texttt{curl}. Hoewel dit effectief is, vereist het meer handmatige configuratie van headers en parameters, wat de integratie met complexe API's kan bemoeilijken.

\item \textbf{Herbruikbaarheid van de code:}
De Python-code is aanzienlijk herbruikbaarder door het gebruik van de standaard packages en de mogelijkheid om logica eenvoudig toe te voegen aan de code. In Rexx is de output vaak sterk afhankelijk van handmatige string-manipulatie (\texttt{parse var}) om data uit een response te halen. Hoewel dit enorm krachtig is, moet er rekening worden gehouden dat indien de structuur van de API-output wijzigt, de Rexx-code ook op meerdere plekken moet worden aangepast. Python's JSON-objecten zijn daarin flexibeler. Voor beide talen zou er kunnen gekozen worden om bepaalde zaken uit de code te modularizeren. Voor de beperkte grootte van de scripts is dat hier niet noodzakelijk. In dit geval zou er voor Rexx wel een en ander ingesteld moeten worden om de scripts goed met elkaar te laten communiceren.

\item \textbf{Security vulnerabilities:}
Er is een aanzienlijk beveiligingsrisico bij de Rexx-implementatie omdat credentials als tekst worden meegegeven in het \texttt{curl}-commando. Dit kan leiden tot lekken in systeemlogs of de terminal geschiedenis. Python biedt veiligere alternatieven zoals de \texttt{getpass} module en een betere afhandeling van HTTPS-certificaten??, hoewel in deze PoC de verificatie voor de private API handmatig is uitgeschakeld voor testdoeleinden. Er wordt natuurlijk wel vanuit gegaan dat de externe Python packages veilig zijn. Aangezien deze packages enorm veel worden gebruikt, kunnen we daar wel degelijk vanuitgaan, maar dat biedt geen 100\% zekerheid.

\item \textbf{Robuustheid van de code:}
Zoals eerder vermeld, als er iets verandert aan de structuur van de API-output, zouden er in Rexx meer zaken moeten veranderen dan in Python, omdat Rexx gebruik maakt van string manipulatie. De Rexx code zal altijd werken zoals de code beschrijft, en zou enkel moeten veranderd worden als de API-output wijzigt. Langs de andere kant kunnen er voor Python veranderingen noodzakelijk zijn als de onderliggende functionaliteit of code van een van de gebruikte packages wijzigt. Het kan dus zijn dat de Python code op een bepaald moment niet langer zal doen wat nu beschreven staat in de code. Dit is een fundamenteel verschil tussen Rexx en Python, waarbij de stabiliteit en voorspelbaarheid van Rexx een groot voordeel is.

\item \textbf{Ondersteuning van de taal:} 
Voor het schrijven van de Python code is weinig werk nodig. Het meest ingewikkelde aspect is het opslaan van de credentials in een tuple. De packages zijn publiek beschikbaar, met een overvloed aan voorbeelden en experten online die informatie geven over het schrijven van de code. Voor de Rexx code daarentegen zijn er minder voorbeelden online. Van IBM is er officiële documentatie over WET, wat veel te complex is voor het simpele voorbeeld in deze use case. Het \texttt{curl} commando wordt vooral aangeraden door gebruikers, en dankzij online documentatie over dit commando, was het schrijven van de Rexx code doenbaar. De samenstelling van het commando door het aaneenplakken van strings is niet vanzelfsprekend en de string manipulatie met de hulp van \texttt{parse var} is, hoewel krachtig, minder intuïtief dan de JSON-objecten in Python.
 
\end{itemize}
